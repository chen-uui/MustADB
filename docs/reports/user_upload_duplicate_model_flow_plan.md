# 用户上传重复命中最小模型/流程设计方案

## 一、当前结构缺口

### 1. `PDFDocument` 当前同时承担两种职责

从现有实现看，`PDFDocument` 既是内容记录，也是审核动作记录：

- 内容记录职责：
  - `filename`
  - `file_path`
  - `file_size`
  - `page_count`
  - `sha1`
  - `processed`
- 审核动作职责：
  - `review_status`
  - `uploaded_by`
  - `reviewed_by`
  - `reviewed_at`
  - `review_notes`

`views_user_upload.py` 在一次用户上传成功后，直接创建 `review_status='pending'` 的 `PDFDocument`。  
`views_review.py` 的待审核列表、通过、拒绝，也都直接围绕 `PDFDocument.review_status` 工作。

### 2. 这为什么阻碍阻断式去重

如果用户上传命中同内容重复后直接“不新建 `PDFDocument`，返回 existing”，就等于同时放弃了：

- 新的待审核队列项
- 本次上传者信息
- 本次上传时间语义
- 本次审核备注承载体

也就是说，当前系统没有“独立审核动作”的第二承载体。  
所以用户上传入口不能像 Weaviate upload 那样，把“复用已有内容记录”和“跳过新动作记录”视为同一件事。

### 3. 如果继续不拆职责，最容易出现的语义问题

最容易出现三类问题：

1. 去重一旦阻断新建 `PDFDocument`，新的用户上传动作就消失
2. 如果仍新建 `PDFDocument` 但复用旧文件，`PDFDocument` 会进一步混合“内容对象”和“审核动作对象”
3. 未来一旦再动审核流、索引流、历史清理，责任边界会更难收口

## 二、候选模型/流程方案对比

### 方案 A：新增 `UploadRequest` / `ReviewRequest` 模型，`PDFDocument` 回归内容记录

数据承载体：

- `PDFDocument`：内容实体
- `UploadRequest` 或 `PDFReviewRequest`：用户上传动作/审核动作实体

是否需要数据库迁移：

- 需要
- 且需要新表、新外键、新状态字段

对用户上传流程的影响：

- 重复命中时，不再重复落盘
- 仍创建新的上传请求记录
- 内容层与动作层彻底分离

对待审核队列的影响：

- 待审核队列将逐步从 `PDFDocument.review_status` 迁到新模型
- 这是结构上最正确的方向

对 `views_review.py` 的影响：

- 高
- 查询、审批、拒绝、列表序列化都要调整

对 `uploaded_by / review_notes / reviewed_by` 语义的影响：

- 更清晰
- 它们将挂在审核动作记录上，而不是挂在内容实体上

对 `file_path / filename / sha1` 语义的影响：

- 更清晰
- 它们只属于内容实体

对前端返回语义的潜在影响：

- 中高
- 即使 API 结构能兼容，返回的底层对象也会变

实施复杂度：

- 高

回归风险：

- 高

是否适合作为“下一轮最小实现”：

- 不适合

判断：

- 这是长期正确方案
- 但对下一轮来说太重，不符合“最小实现”

### 方案 B：不新增模型，在 `PDFDocument` 上增加“复用已有内容”的最小关联

建议形式：

- 在 `PDFDocument` 上新增一个可空自关联字段，例如 `content_source` / `canonical_document` / `duplicate_of`
- 含义是：
  - 为空：该记录自带物理文件，是内容主记录
  - 不为空：该记录是新的审核动作，但复用已有内容记录的文件与哈希

数据承载体：

- 仍然只用 `PDFDocument`
- 通过一条自关联把“审核动作记录”指向“内容主记录”

是否需要数据库迁移：

- 需要
- 但只需要一条可空外键和索引，迁移规模较小

对用户上传流程的影响：

- 用户上传全新 PDF：保持原样
- 用户上传不同名同内容 PDF：
  - 不再重复落盘
  - 新建一条 `review_status='pending'` 的 `PDFDocument`
  - 该记录指向已有内容记录

对待审核队列的影响：

- 仍可继续复用现有 `views_review.py` 主流程
- 新的重复命中上传仍会出现在待审核列表里

对 `views_review.py` 的影响：

- 低到中
- 核心审批动作可基本不变
- 最多只需补充极少数字段展示或日志上下文

对 `uploaded_by / review_notes / reviewed_by` 语义的影响：

- 保持在新建的待审核 `PDFDocument` 上
- 本次上传者与本次审核记录仍可独立保留

对 `file_path / filename / sha1` 语义的影响：

- `sha1` 仍可直接复用
- `file_path` 可直接引用主内容记录路径
- `filename` 仍保留本次上传文件名
- 但模型语义会继续混合：同一物理内容可对应多条 `PDFDocument`

对前端返回语义的潜在影响：

- 低
- 理论上可继续返回新 `document_id`
- 前端不需要立即配合

实施复杂度：

- 中

回归风险：

- 中

是否适合作为“下一轮最小实现”：

- 适合

判断：

- 这是战术性折中方案
- 不是长期最干净的模型
- 但它能在不改前端、不重做审核流的前提下，实现“避免重复落盘 + 保留独立审核动作”

### 方案 C：保留现状，只增加重复命中审计记录/事件记录

数据承载体：

- `PDFDocument` 保持不变
- 额外增加日志/事件记录，或继续只用现有识别日志

是否需要数据库迁移：

- 不一定
- 纯日志版不需要
- 若引入审计表则需要

对用户上传流程的影响：

- 几乎无
- 仍重复落盘、仍重复建待审核记录

对待审核队列的影响：

- 无变化

对 `views_review.py` 的影响：

- 无或极低

对 `uploaded_by / review_notes / reviewed_by` 语义的影响：

- 无变化

对 `file_path / filename / sha1` 语义的影响：

- 无变化

对前端返回语义的潜在影响：

- 无变化

实施复杂度：

- 低

回归风险：

- 低

是否适合作为“下一轮最小实现”：

- 不适合作为本主线的下一轮

判断：

- 它适合继续观察
- 但它没有真正解决“避免重复落盘”

### 方案 D：新增极轻量 `UploadAttempt` 审计表，但不改待审核主体

数据承载体：

- `PDFDocument` 仍承载待审核
- `UploadAttempt` 只记录谁在何时命中了哪个内容

是否需要数据库迁移：

- 需要

对用户上传流程的影响：

- 可在重复命中时不落盘
- 但仍缺少“新的待审核对象”与旧内容之间的清晰审核语义

对待审核队列的影响：

- 弱
- 审计表不是待审核队列

对 `views_review.py` 的影响：

- 低

对 `uploaded_by / review_notes / reviewed_by` 语义的影响：

- 只能保留上传审计，不足以承载完整审核动作

对 `file_path / filename / sha1` 语义的影响：

- 内容层能更稳
- 但审核层仍不完整

对前端返回语义的潜在影响：

- 中
- 需要解释“上传被记录了，但为什么没出现新待审核项”

实施复杂度：

- 中

回归风险：

- 中

是否适合作为“下一轮最小实现”：

- 不适合

判断：

- 它比方案 C 多一步
- 但没有真正解决“保留独立审核动作”

## 三、推荐的最小实现方向

### 推荐方案

当前最推荐的是：**方案 B，给 `PDFDocument` 增加最小自关联，作为短期过渡方案。**

原因：

1. 能满足本主线的两个硬目标：
   - 避免重复落盘
   - 保留独立审核动作
2. 不需要前端立刻改
3. 不需要把 `views_review.py` 全部重写
4. 不需要把 Weaviate 试点语义直接复制到用户上传入口

### 为什么它比方案 A 更适合作为下一轮

方案 A 虽然更干净，但它要求：

- 新增模型
- 迁移待审核队列来源
- 调整审核通过/拒绝的承载对象
- 重新定义前端看到的“PDF 待审核项”是什么

这些都已经超出“下一轮最小实现”。

### 4 个关键设计问题的建议答案

#### 1. 命中重复时，是否仍要给审核员一个新的待办对象？

- **是**

理由：
- 用户上传入口的业务语义就是“提交一个待审核对象”
- 如果没有新待办，用户动作就被静默折叠

#### 2. 这个待办对象是否必须保留本次上传者与上传时间？

- **是**

理由：
- 否则这次上传动作不可追踪
- 审核员也无法知道这是历史旧对象还是新的重复提交

#### 3. 如果复用已有内容文件，新的审核动作应绑定在哪里？

- **短期绑定在新的 `PDFDocument` 记录上**
- 并通过 `duplicate_of` 之类字段指向内容主记录

理由：
- 这样能最大限度复用现有待审核列表和审批逻辑

#### 4. 历史 `approved / rejected / pending` 记录遇到新的重复上传时，系统最稳的默认策略是什么？

- **统一创建新的 `pending` 审核动作，不自动继承历史结论**

理由：
- 最保守
- 不会把“历史审核结论”误当成“本次新动作的审核结论”
- 也避免顺手改审核业务规则

## 五、必须先补的验证

### 1. 内容主记录与审核动作记录的最小判定规则

需要验证：

- `duplicate_of is null` 的记录是否视为内容主记录
- `duplicate_of is not null` 的记录是否只代表一次新的待审核动作

### 2. 审核通过/拒绝 duplicate-hit 记录的副作用边界

需要验证：

- 审核 duplicate-hit 记录时，不会误修改被指向的历史记录状态
- `reviewed_by/reviewed_at/review_notes` 只写在当前动作记录上

### 3. 文件访问兼容性

需要验证：

- 多条 `PDFDocument` 指向同一 `file_path` 时，现有 download/view 逻辑仍正常
- 删除逻辑如果未来存在，不能误删共享文件

### 4. filename guard 不被误伤

需要验证：

- 同名不同内容仍按当前谨慎策略处理
- 不会被错误归类为内容重复

## 六、当前不建议做的事

1. 不要直接把 Weaviate `existing=true` 语义复制到用户上传入口
2. 不要一边去重一边改审核通过自动索引语义
3. 不要一边去重一边做历史重复文件清理
4. 不要同时重构 `PDFDocument` 全模型和审核流
5. 不要在下一轮就引入全新的 `UploadRequest` + 全量 review 流重写
6. 不要修改前端上传成功语义
7. 不要迁移历史 `PDFDocument` 去补 `duplicate_of`

## 七、建议的实施顺序

1. 先按方案 B 做最小迁移设计
2. 只在用户上传入口实现“不同名同内容不重复落盘，但新建 pending 记录”
3. 保持 Weaviate 试点、direct processing、sync/reprocess 逻辑不变
4. 跑最小 smoke 与回归检查
5. 观察是否暴露出 `PDFDocument` 双重职责的明显副作用
6. 如果副作用可控，再评估是否进入更长期的方案 A

## 结论

**结论 A：现在适合进入最小模型/流程实现**

原因：

- 当前最小实现方向已经可以明确收敛到方案 B
- 它只需要一次小型数据库迁移和有限的后端接线
- 不要求前端配合，也不要求重写审核流
- 它不是长期最理想方案，但足够作为下一轮可控、可验证、可回滚的过渡实现

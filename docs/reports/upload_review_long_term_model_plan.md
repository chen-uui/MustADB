# 上传/审核长期模型评估方案

## 一、当前过渡方案评估

### 1. `duplicate_of` 过渡方案的优点

当前 `duplicate_of` 方案已经证明了它适合作为短期过渡层：

- 不改前端
- 不重写审核流
- 不改 Weaviate 试点语义
- 不做历史回填
- 能在用户上传入口实现“不同名同内容不重复落盘，但仍保留新的待审核动作”

它最大的价值是把最关键的物理去重问题先控制住，同时保留了当前产品上更重要的“新的审核动作仍然存在”。

### 2. `duplicate_of` 方案的局限

它的局限也很明确：

1. `PDFDocument` 仍同时承担两类职责
   - 内容记录
   - 审核动作记录
2. 同一份物理内容现在可能对应多条 `PDFDocument`
3. `review_status / uploaded_by / reviewed_by / reviewed_at / review_notes` 仍挂在 `PDFDocument` 上
4. `file_path / sha1 / processed` 仍与审核动作耦合在同一模型里
5. 前端和审核员当前看不到“这是 duplicate-hit 动作”这一层显式语义

### 3. 当前过渡方案还能支撑多久

结论是：

- **它足够支撑当前单入口、短中期过渡**
- 只要以下前提不变，它可以继续用一段时间：
  - duplicate-hit 只在用户上传入口试点
  - 前端暂时不需要显式区分“内容记录”和“审核动作”
  - 审核流仍可接受基于 `PDFDocument.review_status` 工作

但它不适合无限扩展。  
一旦 duplicate-hit 语义要扩到更多入口，或者前端/审核流要理解“动作历史”，`duplicate_of` 就会开始显得吃力。

## 二、长期候选模型对比

### 方案 1：`UploadRequest` / `ReviewRequest` 双模型方案

#### 数据职责边界

- `PDFDocument`
  - 内容主记录
  - `file_path / sha1 / processed / page_count`
- `UploadRequest`
  - 一次上传动作
  - 上传者、上传时间、原始文件名、命中情况
- `ReviewRequest`
  - 一次审核动作或审核轮次
  - 审核状态、审核人、审核时间、审核备注

#### 是否需要迁移

- 需要
- 且是较重迁移
- 需要新表、新外键、新状态流

#### 对审核流的影响

- 高
- `views_review.py` 最终要从“查 `PDFDocument.review_status`”转向“查 `ReviewRequest`”

#### 对前端的影响

- 中高
- 即使 API 可以短期兼容，前端最终也会面临对象语义变化

#### 对 `duplicate_of` 过渡方案的兼容方式

- `duplicate_of` 可以继续短期保留
- 新的 `UploadRequest` 可把 `duplicate_of` 指向的内容主记录作为目标内容
- 但中长期应逐步把 `duplicate_of` 从核心语义降级为迁移桥接字段

#### 实施复杂度

- 高

#### 风险

- 高
- 风险主要来自：
  - 审核流切换
  - 列表查询改写
  - 历史状态兼容

#### 评价

- 这是结构上最完整的长期方案
- 但它太重，不适合作为当前下一步

### 方案 2：`UploadEvent` / `ReviewAction` 轻量动作模型方案

#### 数据职责边界

- `PDFDocument`
  - 仍作为内容主记录
  - 负责文件、哈希、处理状态
- `UploadEvent`
  - 记录每次上传动作
  - 上传者、上传时间、原始文件名、是否 duplicate-hit、命中的主记录是谁
- `ReviewAction`
  - 记录审批/拒绝动作
  - 审核人、动作类型、时间、备注

#### 是否需要迁移

- 需要
- 但比双模型方案轻
- 可以先新增动作表，不立刻迁出全部主流程

#### 对审核流的影响

- 中
- 短期可以继续让 `PDFDocument.review_status` 作为运行中的主状态
- 同时把动作模型作为事件历史与未来切换的铺垫

#### 对前端的影响

- 低到中
- 短期可无感
- 中长期若要展示动作轨迹，再逐步开放字段

#### 对 `duplicate_of` 过渡方案的兼容方式

- 兼容性最好
- `duplicate_of` 仍保留当前语义
- 新增的 `UploadEvent` 可直接记录：
  - 本次动作关联到哪个 `PDFDocument`
  - 是否命中 duplicate-of
- `ReviewAction` 也可先 shadow-write，不打断现有审核流

#### 实施复杂度

- 中

#### 风险

- 中
- 主要风险是双写期需要保持一致性

#### 评价

- 这是最平衡的长期方向
- 它不是一步到位的“完美拆分”
- 但能让系统逐步从“单模型混合语义”迁出，而不推翻当前过渡方案

### 方案 3：继续扩展 `PDFDocument` 的保守方案

#### 数据职责边界

- 所有东西仍尽量挂在 `PDFDocument`
- 继续通过 `duplicate_of`、更多状态字段、更多关联字段来表达动作语义

#### 是否需要迁移

- 需要，但可分散进行

#### 对审核流的影响

- 短期低
- 长期高，因为复杂度会持续堆积

#### 对前端的影响

- 短期低
- 长期会越来越难解释对象语义

#### 对 `duplicate_of` 过渡方案的兼容方式

- 完全兼容
- 因为它本质上就是继续沿这条路往前走

#### 实施复杂度

- 短期低
- 长期高

#### 风险

- 中高
- 主要风险不是立刻坏掉，而是长期可维护性持续下降

#### 评价

- 它是最容易继续写代码的方案
- 也是最容易把语义越堆越混的方案
- 不建议作为长期方向

## 三、推荐长期方向

### 推荐结论

**推荐长期方向：方案 2，先引入轻量 `UploadEvent` / `ReviewAction` 动作模型。**

原因：

1. 它不要求立刻推翻当前 `duplicate_of` 方案
2. 它能把“内容记录”和“动作记录”逐步分开
3. 它不要求立刻重写前端
4. 它允许先 shadow-write，再决定何时把审核列表切过去

### 为什么不直接选方案 1

方案 1 更正统，但当前项目还没有足够强的约束说明必须立刻做全量语义切换。  
在现阶段直接上双模型，会把当前一条可控主线升级成一次中高风险模型重构。

### 为什么不推荐继续走方案 3

继续把 `PDFDocument` 做成“内容 + 上传 + 审核 + duplicate lineage”的全集合，会让：

- 状态字段越来越多
- 视图逻辑越来越难解释
- 未来每次改审核/上传语义都必须碰核心模型

这条路短期省事，长期代价最高。

## 四、如果未来实施，建议的分阶段路径

### Phase 1：只加动作模型，不切主流程

建议内容：

- 新增 `UploadEvent`
- 新增 `ReviewAction`
- 用户上传、approve、reject 时只做 shadow-write
- `PDFDocument.review_status` 仍是当前运行中的主状态

目标：

- 先把动作历史沉淀下来
- 不影响现有前端和审核入口

### Phase 2：让 review API 具备双读能力

建议内容：

- `views_review.py` 继续返回当前兼容结构
- 但内部开始验证：是否可由 `UploadEvent + ReviewAction` 推导出同样的待审核列表

目标：

- 先验证长期模型是否真的足够表达当前业务
- 不急于切换 source of truth

### Phase 3：决定是否切换主语义

只有当以下条件满足时才考虑：

- duplicate-hit 审核动作显著增多
- 前端需要显式展示“本次上传动作”而不是仅展示内容记录
- 更多入口需要统一动作审计

这时才值得考虑把审核流主语义从 `PDFDocument.review_status` 迁向动作模型。

## 五、当前为什么不应该马上重构

### 1. 当前过渡方案刚落地且验证通过

当前 `duplicate_of` 方案已经满足本轮目标：

- 用户上传入口不再重复落盘
- 新的待审核动作仍保留
- 前端无感
- smoke 无回归

这时立刻启动长期模型重构，收益不如风险清晰。

### 2. 还没有足够强的触发信号

目前还缺少这些强信号：

- 多入口都需要动作语义统一
- 审核员明确需要看上传/审核动作轨迹
- 前端需要把 duplicate-hit 记录和内容主记录显式区分

在这些诉求还没变成真实约束前，不值得马上重构。

### 3. 立即重构会同时打开太多面

如果现在立刻做长期模型重构，往往会连带碰到：

- 审核流重写
- 前端语义变化
- 历史数据解释
- duplicate_of 过渡兼容

这与当前“保持已落地过渡方案稳定”是冲突的。

## 六、建议的观察点和触发条件

建议持续观察这些点：

1. duplicate-hit 用户上传的数量是否持续增长
2. 审核员是否开始抱怨“看不清这条 pending 是新内容还是复用内容”
3. 前端是否提出展示 duplicate lineage / 上传历史 / 审核轨迹的需求
4. 是否要把相同语义扩展到 Weaviate 之外的更多入口
5. `PDFDocument` 上是否开始继续堆更多“动作语义字段”

以下情况出现时，才值得正式进入长期模型重构规划：

- duplicate-hit 已不是边缘场景，而是稳定存在的主路径
- 多个入口都需要“内容记录”和“动作记录”分离
- 审核流需要按“动作”而不是按“内容记录”做统计/追踪
- 前端明确要求展示上传/审核事件历史

## 结论

**结论 B：应在后续版本规划长期模型，但现在不要实施**

原因：

- 当前 `duplicate_of` 过渡方案已经足够支撑现阶段目标
- 长期上确实有必要把“内容记录”和“动作记录”拆开
- 但现在还不具备必须立刻重构的压力
- 最合理的做法是先把长期方向规划为轻量 `UploadEvent / ReviewAction` 模型，再等待明确触发条件出现时实施

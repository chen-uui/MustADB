# 用户上传入口重复命中语义设计方案

## 一、问题定义

### 1. 当前用户上传入口的真实语义

`views_user_upload.py` 当前不只是“保存一个 PDF 文件”，而是在一次用户上传成功后同时完成两件事：

1. 把文件落盘到 `backend/media/uploads/`
2. 创建一条新的 `PDFDocument`，并显式写成 `review_status='pending'`

这意味着用户上传入口处理的不是单纯“文件是否重复”，而是两个语义同时存在：

- 文件层语义：是否又引入了一份相同内容的 PDF
- 用户动作语义：是否发生了一次新的、应进入待审核队列的上传动作

### 2. 为什么这和 Weaviate upload 不一样

Weaviate upload 已经存在 `existing=true` 的跳过上传语义，且它的主要职责更接近“把内容送入文档库/处理链路”。  
用户上传入口则不同，它当前天然承担“生成待审核对象”的职责。

所以“同内容重复命中”在用户上传入口里不能只理解为文件重复，还必须回答：

1. 这次上传动作本身是否需要留下独立可追踪痕迹？
2. 如果不新建 `PDFDocument`，审核员还看得到这次新上传吗？
3. 如果仍新建 `PDFDocument`，那去重究竟是在节省磁盘，还是还要保留新的审核动作？

### 3. 当前最关键的结构性约束

从现有模型和流程看，`PDFDocument` 同时承载了以下职责：

- 文件元数据记录
- 待审核/已通过/已拒绝状态
- 上传者 `uploaded_by`
- 审核信息 `reviewed_by/reviewed_at/review_notes`

`views_review.py` 的审核流也是直接围绕 `PDFDocument.review_status` 工作。  
因此如果“同内容重复”时不再新建 `PDFDocument`，就等于默认放弃了一条新的待审核对象，除非系统先引入另一种“上传动作 / 审核请求”承载体。

### 4. 结论化定义

用户上传入口的重复命中，本质上不是纯文件去重问题，而是：

- “同内容文件重复”  
- “新的用户上传动作是否仍应进入审核流程”

这两个问题叠在一起。只解决前者、不定义后者，容易把去重做成语义缺失。

## 二、候选语义方案对比

### 方案 A：完全阻断，直接返回 existing

定义：
- 命中相同 `sha1` 时，不再保存新文件
- 不再新建 `PDFDocument`
- 直接返回已有文档记录，语义类似 Weaviate 试点

影响评估：
- 对用户上传体验的影响：用户会看到“已存在/已跳过”，体验直接，但可能不清楚自己的上传行为是否被系统记录
- 对审核流程的影响：高风险；不会新增待审核项，审核员看不到这次新上传动作
- 对 `PDFDocument` 语义的影响：把 `PDFDocument` 彻底当作“唯一内容记录”，不再承载每次用户上传动作
- 对 `ProcessingTask` / 索引流程的影响：最简单，因为可直接复用既有记录，不新增处理链路
- 对前端提示语义的影响：需要明确 existing/duplicate hit 提示；前端现有“上传成功，等待审核”语义不再准确
- 实施复杂度：低
- 回归风险：高
- 是否需要数据库结构变化：不需要
- 是否适合作为最小下一步：不适合

判断：
- 这是技术上最省事的方案
- 但会直接吞掉“新的待审核动作”
- 不建议直接用于用户上传入口

### 方案 B：不重复落盘，但保留新的待审核动作记录

定义：
- 命中相同 `sha1` 时，不再保存第二份物理文件
- 但仍为这次用户上传保留一条独立的、可审核的动作记录
- 这条记录不一定必须是新的 `PDFDocument`，也可以是新的“上传请求/审核请求”模型

影响评估：
- 对用户上传体验的影响：最符合直觉。用户提交成功，且仍进入审核，只是不再制造重复物理文件
- 对审核流程的影响：最好；审核员仍能看到这次上传请求
- 对 `PDFDocument` 语义的影响：可能需要把“内容记录”和“审核动作记录”拆开
- 对 `ProcessingTask` / 索引流程的影响：可保持稳定，因为内容文件不重复、动作记录独立
- 对前端提示语义的影响：可以保留“已提交审核”，再补充“内容已存在，复用已有文件”
- 实施复杂度：高
- 回归风险：中高
- 是否需要数据库结构变化：大概率需要，或至少需要新模型/新关联
- 是否适合作为最小下一步：当前不适合

判断：
- 这是语义上最完整、长期最健康的方案
- 但它已经超出“最小实现”，因为当前系统没有独立的“上传动作/审核请求”承载模型

### 方案 C：仍新建待审核 `PDFDocument`，但引用已存在文件/sha1

定义：
- 命中相同 `sha1` 时，仍创建新的 `PDFDocument(review_status='pending')`
- 但不再落盘新文件，而是让新记录引用已存在文件路径或共享底层内容

影响评估：
- 对用户上传体验的影响：基本不变，仍能看到新上传进入待审核
- 对审核流程的影响：较稳，待审核语义保留
- 对 `PDFDocument` 语义的影响：一份物理文件对应多条 `PDFDocument`；模型仍然混合“内容对象”和“审核动作对象”
- 对 `ProcessingTask` / 索引流程的影响：中风险；若后续有依赖 `file_path` 唯一性或“一个文档一条文件路径”的隐含假设，容易出问题
- 对前端提示语义的影响：几乎不需要大改
- 实施复杂度：中
- 回归风险：中高
- 是否需要数据库结构变化：不一定，但需要非常仔细验证现有 `file_path/filename` 假设
- 是否适合作为最小下一步：不建议直接上

判断：
- 这是不改审核语义的最短路径
- 但会把 `PDFDocument` 的双重职责进一步固化，后续更难拆
- 同时当前 `filename` 仍是唯一约束，这意味着“不同名同内容”的用户上传如果仍新建记录，命名/引用语义会变得很拧巴

### 方案 D：继续只识别，不阻断；先把用户上传重复命中做成明确语义准备

定义：
- 保持当前用户上传行为不变：仍落盘、仍建新的待审核 `PDFDocument`
- 只继续记录 `duplicate_detected/duplicate_document_id/filename_conflict_same_content`
- 把“未来如果要阻断，到底是阻断文件、阻断记录，还是只改变提示”先设计清楚

影响评估：
- 对用户上传体验的影响：无变化
- 对审核流程的影响：无变化
- 对 `PDFDocument` 语义的影响：无变化
- 对 `ProcessingTask` / 索引流程的影响：无变化
- 对前端提示语义的影响：无变化
- 实施复杂度：低
- 回归风险：低
- 是否需要数据库结构变化：不需要
- 是否适合作为最小下一步：适合继续观察，但不属于真正的“阻断式去重试点”

判断：
- 这是当前最稳的保守路径
- 但它解决的是“识别准备”，不是“用户上传重复命中语义落地”

## 三、当前最稳的推荐方案

### 推荐结论

当前最稳的推荐不是直接落地方案 A，而是：

1. 在产品/流程层先明确“用户上传动作是否必须独立留痕”
2. 如果答案是“必须”，则长期推荐方向更接近方案 B
3. 在没有新动作模型前，不要贸然把方案 A 或方案 C直接落地

### 为什么不是直接推荐方案 A

因为现有系统里，新的待审核对象就是新的 `PDFDocument`。  
如果用户上传命中重复后直接返回 existing：

- 新的 `uploaded_by` 不会写进去
- 新的待审核项不会出现
- 审核员无法区分“旧记录一直在”还是“刚刚又有人重复上传了一次”

这不是单纯的文件优化，而是流程语义变化。

### 为什么方案 B 最符合长期方向

如果未来确实要在用户上传入口做阻断式去重，最合理的目标是：

- 不重复存文件
- 但保留新的用户上传动作
- 审核队列仍可追踪“谁、何时、上传了什么、命中了哪个已存在内容”

这需要把“文件内容实体”和“审核/上传动作实体”分开。  
当前项目还没有这层结构，所以现在只能把它作为推荐方向，而不是立刻实施。

## 四、当前不建议做的方案

### 1. 不建议直接照搬 Weaviate 的 existing 语义

原因：
- Weaviate upload 本来就偏向内容入库/处理入口
- 用户上传本来就偏向“提交待审核对象”
- 两者不是同一业务动作

直接复制 Weaviate 的 `existing=true` 语义，会把“新的用户上传动作”静默折叠进旧记录。

### 2. 不建议一边去重一边改审核语义

例如：
- 重复命中后自动通过审核
- 重复命中后自动复用历史审核状态
- 重复命中后直接跳过待审核

这些都把“存储优化”升级成“审核业务规则改造”，风险明显越界。

### 3. 不建议一边去重一边改索引/处理语义

例如：
- 重复命中后顺手自动索引
- 重复命中后自动跳过后续处理
- 重复命中后把旧 `ProcessingTask` 复用给新动作

这会把去重和处理链路耦合，回归后很难定位是文件判重问题还是业务流程问题。

### 4. 不建议把历史重复清理和新上传命中策略绑在一起

原因：
- “防止新重复继续产生” 与 “清理旧重复” 是两类完全不同的改造
- 前者是入口语义问题
- 后者是历史数据治理问题

如果绑在一起做，验证口径、回滚方式、风险边界都会失控。

### 5. 当前不建议直接落地方案 C

虽然它比方案 A 更保留审核语义，但仍有这些问题：

- `PDFDocument` 继续同时扮演内容实体和审核动作实体
- 多个记录共享一个 `file_path` 后，很多隐含假设需要重新核对
- `filename` 唯一约束与“不同名同内容”的复用策略并不天然顺滑

因此它不像表面看起来那么“最小”。

## 五、如果要实施，必须先补的验证

如果未来真的把用户上传入口从“识别”推进到“阻断式去重”，至少要先补以下验证。

### 1. 用户上传全新 PDF

预期：
- 仍正常落盘
- 仍创建新的 `PDFDocument(review_status='pending')`
- 前端返回仍兼容当前成功语义

### 2. 用户上传同名同内容 PDF

预期：
- 继续由现有 filename guard 处理，或明确升级为更清晰的 duplicate hit 语义
- 不能把现有 filename guard 与内容重复判定混淆

### 3. 用户上传不同名同内容 PDF

这是最关键场景。  
必须先明确未来预期到底是哪一种：

- 返回 existing，不新建待审核对象
- 不落盘，但仍新建新的审核动作记录
- 仍新建 `PDFDocument`，但复用旧文件

不先选定目标语义，不能直接编码。

### 4. 用户上传同名不同内容 PDF

预期：
- 仍按“文件名冲突”谨慎处理
- 绝不能误判为内容重复

### 5. 用户上传后，审核流程仍可用

需要验证：
- 待审核列表是否仍出现正确对象
- 审核通过/拒绝是否仍只作用于正确记录
- `uploaded_by/reviewed_by/reviewed_at/review_notes` 是否仍完整

### 6. 前端返回语义是否仍兼容

需要验证：
- 现有前端是否依赖“上传成功后一定返回新 `document_id`”
- 若未来返回 existing，前端是否会误当作普通成功上传
- 是否需要区分“上传成功并进入审核”与“命中已有内容”

### 7. 是否会误伤已有待审核记录

需要特别验证：
- 当历史上已经存在一条 `pending` 记录，新的重复上传是否应再次形成一个待审核动作
- 当历史上存在 `approved` 或 `rejected` 记录，新的上传是否允许重新进入待审核

这个问题不先定义清楚，任何阻断策略都可能误伤真实业务。

## 六、建议的下一步

### 建议顺序

1. 先做产品/流程确认：用户上传动作是否必须独立留痕
2. 如果答案是“必须”，先补一个最小模型/流程设计
   - 不一定立刻实现
   - 但至少要明确未来是新增“上传请求/审核请求”实体，还是扩展现有 `PDFDocument`
3. 在设计明确前，保持用户上传入口继续停留在“只识别、不阻断”
4. 继续观察 Weaviate 试点是否稳定，再决定是否值得把更复杂语义带入用户上传入口

### 实施前最小设计问题清单

在真正写代码前，至少先回答这 4 个问题：

1. 命中重复时，是否仍要给审核员一个新的待办对象？
2. 新的待办对象是否必须能反映本次上传者与上传时间？
3. 如果复用旧记录，前端是否要看到“命中重复，未新建审核项”？
4. 历史 `approved/rejected/pending` 记录遇到新重复上传时，是否允许重新触发审核？

只要其中任一项未定，直接实施都过早。

## 结论

**结论 B：还需要先补模型/流程设计，再实施**

原因：

- 用户上传入口当前不是单纯文件存储入口，而是“创建待审核对象”的业务入口
- 现有 `PDFDocument` 同时承载内容记录与审核队列语义
- 在没有独立“上传动作/审核请求”承载体前，直接把 Weaviate 的 existing 语义扩到用户上传入口，极易丢失新的审核动作
- 因此下一步不应直接编码，而应先把用户上传重复命中的产品/流程语义定义清楚
